A Brief Discussion on Password Dissection Mechanism

One of my own innovative thinking in the password security ecosystem is the Password Dissection Mechanism. This mechanism has mainly been proposed to establish more control over passwords and the inputs in password fields. The main motivation behind breaking a password string into small pieces comes from the differential calculus’ lim h → 0, where the calculation for a very small change in h can be done, and h → 0 made me realize I have to go small to have the best understanding and best control.

Thus, this dissection mechanism has the capability of tracking every change in every position and in every value, which is absent in the traditional password practice where hashing the string is the ultimate work. In existing mechanisms, the system is totally blind about the password string, and it can only compare the stored password hash with the newly entered password hash during login and tell whether the hash is fully matched or not.

What if the user is benign and making mistakes out of confusion? In my opinion, it happens with everybody, where due to diverse password policies on different platforms, people are forced to set different passwords and forget which platform had which password. After a few tries, they get locked with the suspicion of being some kind of guessing attacker and are ultimately forced to use “Forgot Password.” There starts another hassle, where the user has to use a completely new password that cannot be matched with any previous password. Therefore, a user has to choose a new password or an old one with more modifications, which ultimately brings more problems for that user.

That is why an innovative solution like Password Dissection is a necessity. There, original dissected blocks will be matched with the new entries, and based on that comparison, the system will be able to build a capability of distinguishing benign and malicious users. It will know whether the person attempting to log in is trying to break in or has really forgotten his/her password. There will also be consideration of the kind of typing mistakes the user is making in a password and the history of usage and behavior with previous logins.

However, this mechanism has been improved compared to what I proposed in my last paper on it, and this advanced version will be discussed in detail in my next paper, where I have merged it with another innovative idea of mine — that is, Dynamic Password Policies — to provide a complete password security framework. The example of differential calculus also goes for the next paper as well, where the whole framework will be designed considering as much small data as can be collected to feed into machine learning models. The goal for now is to make a centralized authentication system.
